# Claude Design 更新ガイド

このガイドは、`docs/claude-design/` 配下の CD ファイルを更新する際に、Claude が作成する **CD 更新指示書** と、Naoya が Claude Design セッションで守る **セッション運用ルール** を定めます。

**Claude Design が正典(Source of Truth)である以上、その更新は「Repo/実装に影響する意思決定」であり、通常の Issue 起票フローと同じ厳格さで扱います。**

---

## 1. CD 更新が必要になる 3 分類

UI 改修 Issue 起票前に、Claude は以下を判定します:

| 分類 | 状況 | 対応 |
|---|---|---|
| **A. CD 修正必須** | 実装が CD 準拠を目指すべきだが、CD が古い/不足している | CD 更新指示書作成 → CD 更新 → `docs/claude-design/` 反映 PR → UI 改修 Issue の順で進める |
| **B. CD 意図的乖離** | 実装が CD と意図的に異なる(Phase 一時措置等) | UI 改修 Issue 内に「CD 意図的乖離」セクションで明記、CD は更新しない |
| **C. CD 修正不要** | CD が最新と一致 | UI 改修 Issue のみ起票 |

UI 改修 Issue 起票時、Claude は必ず上記 3 分類のいずれに該当するかを判定した上で起票します(Issue 本文の「改修分類」ブロックに記載)。

## 2. Claude が CD 更新指示書を作成する時のルール

### 2.1 保存先

`Vault (nkhippo/Vault) の 30_projects/<project>/handoff/cd-updates/<YYYY-MM-DD>_<改修概要>.md`

例: `30_projects/IPASoundDrill/handoff/cd-updates/2026-07-23_issue-128-a2-a5-a6.md`

### 2.2 指示書に必ず含める 8 セクション

1. **セッションのゴール**(1 段落、成果物を明示)
2. **背景**(なぜこの CD 修正が必要か、判断の根拠となる Issue / feedback / decision を引用)
3. **保持事項**(このセッションで変更しないもの、除外リストの前段)
4. **修正項目**(各項目に対象セクション ID、意匠仕様、追加/変更/削除の粒度)
5. **影響範囲チェック**(cross-screen impact、画面横断コンポーネント修正時は全該当画面をリスト化)
6. **多言語対応方針**(下記 4.1 のデフォルトに従う、逸脱時のみ明記)
7. **除外リスト**(絶対に触らないファイル/セクション)
8. **セッション終了前のセルフチェック**(具体的なチェック項目リスト、修正項目に対応)

### 2.3 事前確認義務(Claude 側の作業)

指示書作成前に、以下を必ず実施:

- **既存 CD の全ファイル grep**: 修正対象のパーツが他画面にも存在しないか確認(類似パーツの見落とし防止)
- **Vault `design-decisions.md` 精読**: 過去に「現状維持」「意図的乖離」と判断された箇所を触らないか確認
- **Naoya 過去 feedback 整合確認**: userMemories + 直近 Chat での明示的な決定と矛盾しないか確認
- **既存 handoff/cd-updates/ の精読**: 過去の CD 更新指示書と矛盾しないか確認
- **修正規模判定**: 軽微(1-3 項目、部分更新)/ 中規模(4-10 項目)/ 大規模(全面刷新)を明示

## 3. Claude Design セッションが守るルール(指示書に明記する)

Claude が CD 更新指示書内に、以下をセッションへの指示として必ず記載します:

### 3.1 SP / PC 両方修正の明示

すべての修正項目に対して、**SP 版と PC 版の両方**を対象とすることを明示。片方のみ修正するケースは例外(例: PC 固有のホバー状態)であり、その場合は指示書で明示的に SP 対象外と記載。

### 3.2 画面横断パーツの全影響画面修正

ヘッダー、フッター、共通コンポーネント(カード意匠、ピル意匠、モーダル等)を修正する場合、**その要素を持つ全画面を影響範囲チェックセクションでリスト化**し、全画面での一貫修正を義務化。

### 3.3 修正セクション ID を必ず明示

修正対象は `#1a-ja`, `#3a` のような**セクション ID で一意に特定**。「1a 周辺」「トップの下の方」のような曖昧な指定は禁止。

### 3.4 修正しないセクションを明示(除外リスト)

「ついで修正」を防ぐため、除外リストで **絶対に触らないファイル/セクション** を明示。特に:
- 明示的に対象としていない Design System トークン
- 他画面の類似要素
- 既存の意匠パターン

### 3.5 修正粒度を明示

各修正項目に対して以下 4 分類のいずれかを明示:
- **要素追加**: 新規要素を配置
- **文言変更**: テキストのみ変更、構造・意匠は保持
- **意匠変更**: 意匠(色/spacing/border 等)のみ変更、要素と文言は保持
- **削除**: 要素/セクションを削除

意匠変更のつもりが構造変更に発展するリスクを防ぐ。

### 3.6 Design System トークンは変更しない

配色・タイポ・spacing・border-radius 等のトークンは、CD 更新セッションでは変更しない(トークン変更は別プロセス、Design System セッションで扱う)。指示書に必ず「トークンは変更しない」と明記。

### 3.7 部分更新か全体更新か明示

- **部分更新**: 特定の項目のみ変更、既存部分の意匠は完全保持
- **全体更新**: 全面刷新、既存部分も再検討可能

指示書冒頭のメタ情報で明示。

### 3.8 矛盾発生時の対処

セッション中に指示書と CD の既存構造が矛盾する場合、または指示書外の修正が必要そうなケースを発見した場合、**その場で判断せず作業を中断し、Naoya に持ち帰る**。指示書に必ず明記。

### 3.9 セルフチェック実施

セッション終了前に、指示書のセルフチェックリストを全項目確認。未完了項目があればセッションを終えない。

## 4. 多言語対応の既定方針

### 4.1 デフォルト

- **既存 CD に存在する言語のみ更新**(現状は ja / en / ko)
- **zh-Hans / zh-Hant / fil**: 現在 CD セクションがない場合は触らない
- Naoya から明示的に「言語 XX も対応する」と指定された場合のみ拡張

### 4.2 逸脱時

デフォルトから逸脱する場合、指示書の「多言語対応方針」セクションで明示的に対象言語をリスト化し、その理由を記載。

---

## 5. セッション終了後の後続作業(必ずセットで進める)

CD だけ更新して Repo / 実装が追随せず放置される事態を防ぐため、**以下 3 段階を必ずセットで進める**:

```
[1] Claude Design セッションで CD 修正
     ↓
[2] Naoya が修正版 CD ファイルを Chat に投入
     ↓
[3] Claude が Repo 更新用の Cursor 指示書を作成し、新規 Issue で起票
     ↓ Cursor が docs/claude-design/ を更新 → PR → merge
[4] Claude が update-log.md 更新を含めることを Cursor 指示書に明記
     ↓
[5] 関連する UI 改修 Issue に「CD 更新完了、実装再開可」コメントを Claude が投稿
     ↓ Cursor/Codex が UI 改修 Issue の実装再開
```

**[3] の Repo 更新 Issue はテンプレート化されており、Claude が自動的に起票**(手動作業なし)。

## 6. 過去の指示書との矛盾チェック

Claude は CD 更新指示書作成前に、以下を実施:

1. `30_projects/<project>/handoff/cd-updates/` 配下の過去指示書を list_directory で取得
2. 直近 3-5 件の指示書を get_file_content で精読
3. 今回の修正が過去の判断と矛盾しないか確認
4. 矛盾があれば指示書に「過去判断との差分」セクションを追加し、Naoya 判断を明示的に仰ぐ
